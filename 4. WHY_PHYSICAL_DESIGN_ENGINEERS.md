# Why Physical Design Engineers Still Matter (Even Though OpenLane "Just Worked")

After successfully running the Half Adder through OpenLane 2 — RTL in, GDS out, no manual intervention — an obvious question came up:

> **If the flow completes automatically, why do companies need Physical Design engineers at all?**

This document is the answer to that question, plus the follow-on project (an 8-bit ALU) that was picked specifically to start feeling *why* the answer is true, instead of just reading about it.

---

## The Short Answer

> **OpenLane automates the physical design *flow*. It does not automate physical design *engineering*.**

Running `flow.tcl` (or `openlane --design-dir . config.json`) executes a sequence of steps. It does **not** think, judge trade-offs, or make engineering decisions. For a 2-gate half adder, the default settings are good enough to get through the whole flow without hitting a single hard problem. That's not proof the flow doesn't need engineers — it's proof the half adder was too small to expose why it does.

---

## What Actually Happened With the Half Adder

```
RTL (half_adder.v)
      │
      ▼
Logic Synthesis
      │
      ▼
Floorplan
      │
      ▼
Placement
      │
      ▼
Clock Tree   (barely relevant — combinational design)
      │
      ▼
Routing
      │
      ▼
DRC / LVS
      │
      ▼
GDSII
```

Everything passed. But the design was **2 logic gates and 4 standard cells total**.

**Analogy:** this is like testing whether Google Maps works by driving once around your own neighborhood. It'll work fine — that doesn't mean the app has been proven at the scale of an entire country's road network.

A real chip — an Apple M-series chip, an NVIDIA GPU — isn't 4 standard cells. It's **500 million to 10+ billion transistors**. At that scale, problems show up that a default flow run cannot solve on its own.

---

## Physical Design Is Fundamentally an Optimization Problem

A manager doesn't just say "make a chip." They say things like:

> "This chip must run at 2.5 GHz."

And the tool might respond:

```
Timing failed.
Worst Negative Slack (WNS) = -420 ps
```

OpenLane reporting this number is not the same as OpenLane *fixing* it. Below are the categories of problems that show up on real designs, each with a real example of the kind of decision a human engineer has to make.

---

### 1. Timing Closure

Example timing path:

```
ALU → Multiplier → Register
```

| | |
|---|---|
| Actual delay | 0.63 ns |
| Required delay | 0.40 ns |
| **Needed improvement** | **230 ps** |

Possible fixes an engineer could choose from:

- Upsize cells on the critical path
- Resize/insert buffers
- Change placement of the involved cells
- Reduce routing congestion around that path
- Restructure the RTL itself (e.g. add a pipeline stage)
- Add useful clock skew

**Key point:** OpenLane can tell you the slack is negative. It cannot decide *which* of these six approaches is the right trade-off for your specific chip, schedule, and power budget. That decision requires engineering judgment.

---

### 2. Congestion

Visualized as a routing map where every track is full:

```
████████████
████████████
████████████
```

Router response:

```
Cannot finish routing
```

Possible fixes:

- Change the floorplan
- Move macros to relieve pressure in a hot area
- Reduce core utilization (give the router more breathing room)
- Add dedicated routing channels
- Change the placement strategy/effort

---

### 3. Power

Example constraint:

| | |
|---|---|
| Power budget | 200 mW |
| Actual power | 410 mW |
| **Needed reduction** | **~50%** |

Possible fixes:

- Multi-Vt cell swapping (trade speed for leakage power)
- Clock gating
- Power gating unused blocks
- Cell resizing
- Buffer/CTS optimization

Automation follows the constraints it's given — it does not invent a power architecture for you.

---

### 4. IR Drop (Power Grid Integrity)

| | |
|---|---|
| Measured voltage drop | 190 mV |
| Allowed voltage drop | 50 mV |

Possible fixes:

- Thicker power grid straps
- More straps overall
- More vias between power layers
- Relocate macros away from weak grid regions

---

### 5. DRC (Design Rule Checks)

Example:

```
1200 DRC errors
```

The flow simply stops and reports this. An engineer then has to investigate categories like:

- Antenna violations
- Spacing violations
- Notch errors
- Via enclosure issues
- Pin access problems

Fixing 1200 violations is not something OpenLane decides how to do — it's a diagnostic and iterative repair process.

---

### 6. Large SoC Floorplanning

The half adder has 4 cells. A modern SoC has:

```
CPU · GPU · AI Accelerator · PCIe · USB · DDR · SRAM · PLL · ADC · DAC
```

...often hundreds of macros. Someone has to decide, using engineering judgment and knowledge of signal flow, thermal behavior, and power distribution:

- Where should the CPU physically sit?
- Where should SRAM go relative to it?
- Where should the PLL (clock source) sit?
- How should the power grid be routed across the whole die?
- Where does the clock enter the chip?
- How much whitespace is left for future routing/fixes?
- Which macros need to be rotated or mirrored?
- How many separate voltage domains does the chip need?

**This entire category of decision *is* physical design.** It has no "default" that works for an arbitrary large chip.

## Why the Half Adder "Just Worked"

The half adder's `config.json` was intentionally minimal:

```json
{
    "DESIGN_NAME": "half_adder",
    "CLOCK_PORT": null
}
```

OpenLane silently filled in the rest using defaults. But OpenLane 2 actually exposes **over 1,200 configuration variables** — things like:

- Core utilization
- Aspect ratio
- Target density
- CTS strategy
- Which routing layers to use
- PDN strap width
- Buffer insertion strategy
- Placement effort
- Timing effort
- Number of repair iterations

A physical design engineer tunes these deliberately, based on what the reports say after each run — not by guessing up front.

## What the Real Workflow Looks Like in Industry

Nobody in a company runs the flow once and calls it done. The real loop looks like:

```
Run flow
   │
   ▼
Timing fails
   │
   ▼
Analyze reports
   │
   ▼
Modify constraints
   │
   ▼
Run again
   │
   ▼
Power too high
   │
   ▼
Optimize (Vt swap / gating / sizing)
   │
   ▼
Run again
   │
   ▼
Congestion detected
   │
   ▼
Modify floorplan
   │
   ▼
Run again
   │
   ▼
IR drop too high
   │
   ▼
Modify power grid (PDN)
   │
   ▼
Run again
   │
   ▼
Clock skew issue
   │
   ▼
Optimize CTS
   │
   ▼
Run again
   │
   ▼
ECO (Engineering Change Order)
   │
   ▼
Run again
   │
   ▼
Signoff
```

On a large chip, **this loop can repeat dozens or even hundreds of times.** The half adder needed zero iterations of this loop — which is exactly why it didn't teach any of these lessons on its own.

---

## The Real Takeaway

> A tiny design (a handful of gates) can be solved almost perfectly by default heuristics.
> A real design (ALUs, multipliers, CPUs, SoCs) starts exposing timing, congestion, power, and routing problems that require human analysis and tuning.

The goal going forward: stop treating OpenLane as a "one-click RTL-to-GDS button," and start treating it as a **laboratory** for investigating *why* things happen. Useful questions to start asking on every future run:

- Why is the worst negative slack (WNS) negative?
- Why did placement density change after I modified utilization?
- What caused this specific routing congestion?
- How did changing cell sizing affect both timing *and* power?
- Why did clock tree synthesis insert this many buffers?
- What happens if I change the floorplan aspect ratio or PDN configuration?

These are the actual day-to-day investigations a physical design engineer performs.

## Next Project: 8-bit ALU (`alu8`)

A half adder can't expose any of the problems above — it's too small. The next design picked was a small **8-bit ALU**, still simple enough to be manageable, but complex enough to start seeing real timing/congestion behavior later.

This is where a fresh setup mistake happened — documented below because it's an easy one to repeat.

### ⚠️ Mistake: RTL file not found

Running the flow gave:

```
Path provided for variable 'VERILOG_FILES[0]' is invalid:
'./alu8.v' does not exist
```

**What this error actually means:** OpenLane hasn't started synthesis yet — it's still just validating the config, and it can't locate the Verilog file listed in `VERILOG_FILES`. This is a good, early failure — much easier to fix than a mid-flow crash.

### Step-by-step debug process used

**1. Confirm what's actually in the project directory:**

```bash
pwd
ls -l
```

Expected to see something like:

```
config.json
alu8.v
```

**2. Check the config file itself:**

```bash
cat config.json
```

**3. Check for filename mismatches** (capitalization, extension, typos):

```bash
ls
```

If the actual file was, say, `ALU8.v` instead of `alu8.v`, either:

```bash
mv ALU8.v alu8.v
```

or update `config.json` to match the real filename exactly. Linux filenames are case-sensitive — `ALU8.v` and `alu8.v` are two different files as far as the OS is concerned.

### The actual root cause

The project's own setup guide specified this folder structure:

```
alu8/
├── config.json
└── src/
    └── alu8.v
```

**The RTL was supposed to live inside `src/`, not the project root.** But `config.json` was pointing at the root:

```json
"VERILOG_FILES": [
    "dir::alu8.v"
]
```

instead of:

```json
"VERILOG_FILES": [
    "dir::src/alu8.v"
]
```

`dir::` is a path *relative to the project directory* — so `dir::alu8.v` looks for the file directly inside the project folder, while `dir::src/alu8.v` looks one level down inside `src/`. The path in the config has to match wherever the file physically is.

### The Fix

**Option 1 (recommended) — move the file to match the expected structure:**

```bash
mkdir -p src
mv alu8.v src/
```

Then update `config.json`:

```json
{
    "DESIGN_NAME": "alu8",
    "VERILOG_FILES": [
        "dir::src/alu8.v"
    ],
    "CLOCK_PORT": null
}
```

**Option 2 — keep the RTL in the project root instead**, and just make sure it's actually there:

```bash
ls
```

should show both:

```
alu8.v
config.json
```

### Reusing the Half Adder's PDN fix, preemptively

Since the ALU is also a purely combinational design (same category as the half adder), it was set up from the start with the same oversized floorplan that fixed the earlier `PDN-0185` strap error — instead of waiting to hit that error again:

```json
{
    "DESIGN_NAME": "alu8",
    "VERILOG_FILES": [
        "dir::src/alu8.v"
    ],
    "CLOCK_PORT": null,
    "FP_SIZING": "absolute",
    "DIE_AREA": [0, 0, 300, 300],
    "CORE_AREA": [20, 20, 280, 280]
}
```

### Verifying the directory structure before rerunning

Before running the flow again, the project structure was double-checked with:

```bash
tree
```

or, if `tree` isn't installed:

```bash
find .
```

This confirms the actual on-disk layout matches what `config.json` expects — cheaper to check up front than to debug a second path error.

## Common Errors — Quick Reference

### `Path provided for variable 'VERILOG_FILES[0]' is invalid: '<file>' does not exist`
**Cause:** the path in `config.json`'s `VERILOG_FILES` doesn't match where the `.v` file actually is on disk (wrong folder, wrong filename, or wrong case).
**Fix:** run `ls -l` / `tree` to see the real layout, then either move the file or correct the `dir::` path in `config.json` to match.

### File "not found" even though a similarly-named file exists
**Cause:** case sensitivity (`ALU8.v` ≠ `alu8.v`) or a typo.
**Fix:** `ls` to see the exact filename, then rename or correct the config to match exactly.

### Flow completes instantly for a small design with zero manual tuning
**Not actually an error** — but a signal to remember: this only happens because the design is small enough for default heuristics to handle. It is not evidence that larger designs won't need engineering intervention.

## Key Concepts From This Session

| Concept | Meaning |
|---|---|
| **Flow automation vs. engineering** | Running steps in sequence is not the same as making design trade-off decisions |
| **WNS (Worst Negative Slack)** | How far the worst timing path misses its required speed — a report, not a fix |
| **Congestion** | Routing tracks in a region are oversubscribed; requires floorplan/placement changes to resolve |
| **IR Drop** | Voltage lost across the power grid due to resistance; fixed via stronger power distribution |
| **ECO (Engineering Change Order)** | A targeted, late-stage fix applied after most of the design is already complete |
| **`dir::` prefix in OpenLane config** | Tells OpenLane the following path is relative to the project directory |
